# 트러블슈팅 항목 작성 형식

커밋 메시지, 코드 주석, CHANGELOG에서 트러블슈팅 소재를 추출하고 작성하는 방법.

---

## 소재 발굴 우선순위

### 1순위: CHANGELOG.md / HISTORY.md
`Fixed`, `Bug`, `Breaking Change` 섹션을 먼저 확인한다.

### 2순위: Git 커밋 메시지
```bash
git log --oneline --all | grep -iE \
  "fix|bug|hotfix|issue|resolve|workaround|revert|refactor|perf|optimize" \
  | grep -v "node_modules" | head -30
```

키워드별 의미 해석:
- `fix:` / `bugfix:` → 버그 수정 → 트러블슈팅 소재
- `revert:` → 잘못된 결정 수정 → 의사결정 회고 소재
- `perf:` / `optimize:` → 성능 개선 → 기술적 도전 소재
- `refactor:` → 구조 개선 → 아키텍처 관련 소재

### 3순위: 코드 내 주석
```bash
grep -rn "FIXME\|HACK\|WORKAROUND\|NOTE:\|TODO:" \
  --include="*.ts" --include="*.tsx" --include="*.js" \
  --include="*.py" --include="*.go" --include="*.java" \
  --exclude-dir=node_modules --exclude-dir=.git \
  | grep -v "TODO:" | head -20
```
`HACK`, `WORKAROUND` 주석은 특히 트러블슈팅 소재로 적합하다.

### 4순위: 의존성 버전 변경
```bash
git log --oneline --all -- package.json requirements.txt go.mod | head -10
```
라이브러리 버전을 올리거나 교체한 커밋이 있다면 호환성 이슈 소재일 수 있다.

## 항목 작성 형식

```markdown
### {문제를 명확히 표현한 제목}
{예: "Next.js App Router에서 SSE 응답 스트리밍이 끊기는 문제"}
{예: "Docker 컨테이너 간 네트워크 통신 실패"}
{예: "페이지네이션 쿼리의 N+1 문제"}

**상황**  
{언제, 어디서, 어떤 증상이었는지. 1~2문장.}

**원인**  
{무엇이 문제였는지. 기술적으로 구체적으로.}

**해결**  
{어떻게 고쳤는지. 가능하면 변경 전/후 대비.}

**배운 점**  
{이 경험에서 얻은 것. 일반화 가능한 인사이트 1~2줄.}
```

## 좋은 예시 vs 나쁜 예시

### ✅ 좋은 예시

```markdown
### JWT 토큰 만료 시 무한 리프레시 루프 발생

**상황**  
액세스 토큰 만료 시 리프레시 요청을 보내는 인터셉터가 있었는데,
리프레시 API 자체가 실패할 경우 인터셉터가 재귀적으로 계속 호출되었다.

**원인**  
Axios 인터셉터에서 리프레시 요청 실패 케이스를 분기하지 않아
모든 401 응답에 동일하게 재시도 로직이 적용되었다.

**해결**  
리프레시 엔드포인트 URL을 화이트리스트로 관리하여
해당 URL에서 발생하는 401은 인터셉터를 통과하지 않도록 수정.

**배운 점**  
인터셉터처럼 전역에서 동작하는 로직은 반드시 자기 자신이 유발하는
무한 루프 케이스를 명시적으로 차단해야 한다.
```

### ❌ 나쁜 예시

```markdown
### 성능 문제 해결

**상황**  
앱이 느렸다.

**원인**  
코드가 비효율적이었다.

**해결**  
코드를 최적화했다.

**배운 점**  
항상 성능을 고려해야 한다.
```

## 항목 수 기준

| 프로젝트 규모 (커밋 수) | 적정 항목 수 |
|------------------------|-------------|
| ~50 커밋 | 1~2개 |
| 50~200 커밋 | 2~3개 |
| 200+ 커밋 | 3~5개 |

소재가 없으면 억지로 채우지 않는다. 항목이 1개여도 괜찮다.  
소재가 아예 없으면 섹션 전체를 생략한다.

---

## 근거 없이 작성하면 안 되는 것

- 실제 에러 메시지 없이 "OOM 에러가 발생했다"
- 커밋/코드에서 확인되지 않은 "배포 장애가 있었다"
- 수치 없이 "응답 속도가 크게 개선되었다"

이런 내용이 README나 CHANGELOG에 있으면 기재 가능. 없으면 작성 금지.
